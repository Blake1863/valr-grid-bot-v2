### 2-Layer Rotation System

**Layer 1 (Account Swap):** Every 3 cycles
- Phase 0: CM1 = maker, CM2 = taker
- Phase 1: CM2 = maker, CM1 = taker

**Layer 2 (Side Swap):** Every 9 cycles
- Phase 0: Maker = SELL, Taker = BUY
- Phase 1: Maker = BUY, Taker = SELL

**Formula:**
```rust
account_phase = (cycle_count / 3) % 2
side_phase = (cycle_count / 9) % 2
```

**Full 18-cycle pattern (~4.5 min at 15s/cycle):**
| Cycles | Maker | Taker |
|--------|-------|-------|
| 0-2 | CM1 SELL | CM2 BUY |
| 3-5 | CM2 SELL | CM1 BUY |
| 6-8 | CM1 SELL | CM2 BUY |
| 9-11 | CM2 BUY | CM1 SELL |
| 12-14 | CM1 BUY | CM2 SELL |
| 15-17 | CM2 BUY | CM1 SELL |

**Issue 1: Query string in signature**

## CM Bot v2 — STP Error Fix (22:53)

### Problem

**net_positions.py cron was using old direct API keys** (CM1_API_KEY, CM2_API_SECRET hardcoded) instead of main key + subaccount impersonation. When it placed SELL on CM1 and BUY on CM2 at the same price, both orders authenticated as the same account context, triggering -11500.

### Critical Clarification
**VALR HAS NO SELF-TRADE PREVENTION ACROSS SUBACCOUNTS.** Cross-subaccount trades ALWAYS execute. If you see -11500, it means BOTH orders are on the SAME account (same auth context), NOT a platform restriction.

## VALR API Authentication — Final Reference

### HMAC-SHA512 Signature
```
message = timestamp + METHOD + path + body + subaccount_id
signature = HMAC-SHA512(API_SECRET, message)
```

### Headers Required
```
X-VALR-API-KEY: <64-char API key>
X-VALR-SIGNATURE: <hex-encoded HMAC>
X-VALR-TIMESTAMP: <Unix ms>
X-VALR-SUB-ACCOUNT-ID: <subaccount ID or omit if main account>
```

### WebSocket Auth (Account Channel)
```
path = "/ws/account"
message = timestamp + "GET" + path
signature = HMAC-SHA512(secret, message)
URL: wss://api.valr.com/ws/account?subaccountId=<ID>&signature=<SIG>&timestamp=<TS>
```

### Critical Rules
1. Method MUST be uppercase (GET, POST, DELETE)
2. Timestamp in milliseconds (not seconds)
3. Subaccount ID must be in signature message string (even if empty)
4. Query strings are part of the path for signature
5. **NO STP across subaccounts** — -11500 always means same-account collision
