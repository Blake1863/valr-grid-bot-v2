### Real root cause ‚ÄĒ WebSocket architecture issues

After reviewing `src/price_feed.rs`, `src/ws_client.rs`, `src/cycle.rs` and comparing to the working grid-bot in `bots/valr-common/src/stream.rs`:

1. **Wrong price-feed event (BIGGEST ISSUE):** `cm-bot-v2` subscribes to `AGGREGATED_ORDERBOOK_UPDATE` (throttled/aggregated, 100‚Äď500ms cadence). Working grid-bot uses `OB_L1_DIFF` (tick-by-tick top of book with `OB_L1_SNAPSHOT` on connect). WS is "healthy" but its view of the book is lagged by the aggregation interval ‚Üí mid price computed is stale ‚Üí maker crosses real book ‚Üí post-only reject ‚Üí taker IOC hits empty price ‚Üí IOC reject.

2. **`PLACE_LIMIT_WS_RESPONSE` is not "order on book".** `ws_client.rs::handle_message` resolves the `place_order()` future on `PLACE_LIMIT_WS_RESPONSE`, which is just "command received". The bot then immediately fires the taker IOC before the maker is actually resting. Real "on book" = `ORDER_STATUS_UPDATE` with `orderStatusType=Placed`.

3. **Account WS only subscribes to `ORDER_STATUS_UPDATE`.** Missing `NEW_ACCOUNT_TRADE` and `OPEN_ORDERS_UPDATE`. That's why `external_fills` stays 0 even though real fills are leaking out ‚ÄĒ bot literally can't see them.

4. **No staleness guard.** `OrderbookPrices` has no `Instant` timestamp; bot trades on whatever's in the map regardless of age.

5. **Spread fallback threshold too high.** `cycle.rs` falls back to mark price only when spread > 50bps, but BTC/ETH/SOL/AVAX perp spreads are 6‚Äď24bps ‚Üí always use (stale) mid.

### Fix plan (not yet implemented ‚ÄĒ awaiting user approval)
Priority order:
- **A.** Switch price feed `AGGREGATED_ORDERBOOK_UPDATE` ‚Üí `OB_L1_DIFF` (single biggest win).
- **B.** Stamp each `OrderbookPrices` with `Instant::now()`; skip cycle if age > 200ms.
- **C.** Make `WsClient::place_order()` resolve on `ORDER_STATUS_UPDATE{Placed}`, not `PLACE_LIMIT_WS_RESPONSE`. This serialises maker‚Üítaker correctly.

User: "Do all the above." Implemented A+B+C+D+E plus a couple of extra fixes discovered during implementation.

### Regression timeline (from log sampling)
Success rate was **healthy until ‚Čą20% April 8** (83-99%), then cratered to **<2% from April 9 onward**. Likely a VALR-side change (matching-engine or aggregation cadence) since cm-bot-v2 source hadn't changed in that window.

### What actually worked
Implementation went through three iterations before stabilising:

1. **First attempt:** OB_L1_DIFF per-pair + `ORDER_STATUS_UPDATE`-only resolution.
   - Broke: OB_L1_DIFF sends *deltas* (with qty=0 = "remove level"), not full L1 snapshots. Our naive parser treated every delta as a full snapshot ‚Üí bid/ask drifted to stale, invalid values (e.g. feed reported SOL bid 87.65 while REST showed 87.97).
   - Also broke: VALR doesn't emit any ORDER_STATUS_UPDATE when a post-only maker *rests* successfully ‚ÄĒ only on terminal states (Filled/Failed/Cancelled/Expired/Partially Filled). So waiting for a status update hung for 10s on every successful maker.

2. **Second attempt:** Kept OB_L1_DIFF; added a 250ms grace timer after PLACE_LIMIT_WS_RESPONSE for makers ‚ÄĒ if no Failed status arrives in that window, assume resting.
   - Partially worked (makers + takers started pairing), but OB_L1_DIFF staleness from issue #1 was still poisoning prices. Cycles that *did* execute were filling against external liquidity at wrong prices.

3. **Third attempt (current, working):** Reverted price feed to `AGGREGATED_ORDERBOOK_UPDATE` (sends full top-of-book per message), kept grace-timer approach for maker confirmation, kept `ORDER_STATUS_UPDATE` handling for taker (IOC always emits terminal).

### Final code structure
- `src/price_feed.rs`: `AGGREGATED_ORDERBOOK_UPDATE` feed (all pairs on one WS); `MARKET_SUMMARY_UPDATE` for mark-price metadata (second WS). `OrderbookPrices` now stamped with `Instant` + `prev_updated_at` for age + jitter checks.
- `src/ws_client.rs`: Actor-based WS client. `PlaceMode::Maker`/`Taker`. On ACK: for maker, spawn 250ms grace timer that resolves Ok if no Failed arrives; for taker, wait for ORDER_STATUS_UPDATE. Both paths share `pending_by_order_id` map keyed by VALR orderId post-ACK. Subscribes to ORDER_STATUS_UPDATE + NEW_ACCOUNT_TRADE + OPEN_ORDERS_UPDATE.
- `src/cycle.rs`: freshness guard (`MAX_ORDERBOOK_AGE = 15s`), jitter guard (`MIN_STABLE_WINDOW = 20ms`), **inside-spread maker pricing**: maker BUY at `bid + tick`, maker SELL at `ask - tick`. Fallback to resting at best-bid/ask only when spread ‚Č§ 1 tick. Removed the old "fallback to mark price when spread > 50bps" logic because it placed makers far from top-of-book and let takers route through external liquidity.

### Measured outcome
