### Real root cause — WebSocket architecture issues

After reviewing `src/price_feed.rs`, `src/ws_client.rs`, `src/cycle.rs` and comparing to the working grid-bot in `bots/valr-common/src/stream.rs`:

1. **Wrong price-feed event (BIGGEST ISSUE):** `cm-bot-v2` subscribes to `AGGREGATED_ORDERBOOK_UPDATE` (throttled/aggregated, 100–500ms cadence). Working grid-bot uses `OB_L1_DIFF` (tick-by-tick top of book with `OB_L1_SNAPSHOT` on connect). WS is "healthy" but its view of the book is lagged by the aggregation interval → mid price computed is stale → maker crosses real book → post-only reject → taker IOC hits empty price → IOC reject.

2. **`PLACE_LIMIT_WS_RESPONSE` is not "order on book".** `ws_client.rs::handle_message` resolves the `place_order()` future on `PLACE_LIMIT_WS_RESPONSE`, which is just "command received". The bot then immediately fires the taker IOC before the maker is actually resting. Real "on book" = `ORDER_STATUS_UPDATE` with `orderStatusType=Placed`.

3. **Account WS only subscribes to `ORDER_STATUS_UPDATE`.** Missing `NEW_ACCOUNT_TRADE` and `OPEN_ORDERS_UPDATE`. That's why `external_fills` stays 0 even though real fills are leaking out — bot literally can't see them.

4. **No staleness guard.** `OrderbookPrices` has no `Instant` timestamp; bot trades on whatever's in the map regardless of age.

5. **Spread fallback threshold too high.** `cycle.rs` falls back to mark price only when spread > 50bps, but BTC/ETH/SOL/AVAX perp spreads are 6–24bps → always use (stale) mid.

### Fix plan (not yet implemented — awaiting user approval)
Priority order:
- **A.** Switch price feed `AGGREGATED_ORDERBOOK_UPDATE` → `OB_L1_DIFF` (single biggest win).
- **B.** Stamp each `OrderbookPrices` with `Instant::now()`; skip cycle if age > 200ms.
- **C.** Make `WsClient::place_order()` resolve on `ORDER_STATUS_UPDATE{Placed}`, not `PLACE_LIMIT_WS_RESPONSE`. This serialises maker→taker correctly.
- **D.** Subscribe to `NEW_ACCOUNT_TRADE` so external fills become visible.
- **E.** (Optional) jitter check: if bid/ask changed in last 50ms, skip cycle.

Rejected idea: "Use GTC + GTC then cancel unfilled" — user says one of them will be cancelled anyway, so it doesn't help.

Insufficient-balance issue (46% of failures) is a separate problem, not the focus of this thread.
